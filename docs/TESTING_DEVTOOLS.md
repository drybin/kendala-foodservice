# Инструкция: проверка логики через `?mockTime`

Эта инструкция показывает, как протестировать блокировку дней,
подменив время через URL-параметр. Работает и на сервере (SSR), и на клиенте.

## 1) Добавьте параметр `mockTime` в URL

Откройте приложение и добавьте `?mockTime=<ISO-дата>` в адресную строку:

```
http://localhost:3000/?mockTime=2026-04-06T13:00:00
```

Или через консоль DevTools:

```js
// Перенаправить на страницу с подменённым временем
location.search = "?mockTime=2026-04-06T13:00:00"
```

> ⚠️ **Важно:** не используйте `location.reload()` — параметр уже в URL, просто перезагрузите страницу обычным способом (F5) или кликните на адресную строку и нажмите Enter.

## 2) Сценарии для проверки

| Сценарий | URL |
|----------|-----|
| Понедельник 13:00 (сегодня доступен) | `?mockTime=2026-04-06T13:00:00` |
| Понедельник 15:00 (сегодня блокируется) | `?mockTime=2026-04-06T15:00:00` |
| Суббота 10:00 (все Пн–Пт заблокированы) | `?mockTime=2026-04-11T10:00:00` |
| Воскресенье 10:00 (все Пн–Пт заблокированы) | `?mockTime=2026-04-12T10:00:00` |

При активном мок-времени на странице появится **жёлтый баннер** с указанием текущего подменённого времени.

## 3) Верните реальное время

Удалите `?mockTime=...` из URL и перезагрузите страницу:

```js
location.search = ""
```

---

## Почему старый вариант с Console не работал

Предыдущая инструкция просила подменить `Date` через консоль DevTools и вызвать `location.reload()`.
Это не работает, потому что Next.js выполняет серверный рендеринг (SSR) **до** того, как браузерный
консольный мок успевает сработать. При перезагрузке сервер использует реальное время, а не ваш мок.

Новый подход с `?mockTime=` читает параметр из URL, поэтому работает одинаково и на сервере, и на клиенте.
